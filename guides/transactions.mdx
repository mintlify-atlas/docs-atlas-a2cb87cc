Datacave supports basic transaction control statements (`BEGIN`, `COMMIT`, `ROLLBACK`) for grouping multiple SQL statements into logical units. However, the current implementation has important limitations regarding atomicity and isolation.

## Transaction Basics

A transaction groups multiple SQL statements together:

```sql
BEGIN;
INSERT INTO accounts VALUES (1, 'Alice', 1000.00);
INSERT INTO accounts VALUES (2, 'Bob', 500.00);
COMMIT;
```

All statements between `BEGIN` and `COMMIT` are treated as a single unit.

## Transaction Commands

### BEGIN / START TRANSACTION

Start a new transaction:

```sql
BEGIN;
-- or
START TRANSACTION;
```

After `BEGIN`, the connection enters transaction mode. The server's `ReadyForQuery` status changes from `Idle` (`I`) to `InTransaction` (`T`).

<Note>
`BEGIN` and `START TRANSACTION` are equivalent in Datacave. Both start a transaction and enable statement buffering.
</Note>

### COMMIT

Commit the current transaction, applying all buffered changes:

```sql
BEGIN;
INSERT INTO users VALUES (1, 'Alice', 'alice@example.com');
INSERT INTO users VALUES (2, 'Bob', 'bob@example.com');
COMMIT;  -- Both inserts are executed
```

When you execute `COMMIT`:

1. All buffered mutating statements (INSERT, UPDATE, DELETE) are executed
2. Changes are written to storage
3. The transaction ends and the connection returns to `Idle` state

### ROLLBACK

Discard the current transaction, abandoning all buffered changes:

```sql
BEGIN;
INSERT INTO users VALUES (1, 'Alice', 'alice@example.com');
INSERT INTO users VALUES (2, 'Bob', 'bob@example.com');
ROLLBACK;  -- Neither insert is executed
```

When you execute `ROLLBACK`:

1. All buffered statements are discarded
2. No changes are written to storage
3. The transaction ends and the connection returns to `Idle` state

## Buffering Behavior

Datacave implements transaction support through statement buffering:

<Steps>
  <Step title="BEGIN starts buffering">
    When you execute `BEGIN`, Datacave starts buffering mutating statements (INSERT, UPDATE, DELETE):
    
    ```sql
    BEGIN;
    INSERT INTO products VALUES (1, 'Widget', 10.99);
    UPDATE products SET price = 12.99 WHERE id = 1;
    DELETE FROM products WHERE id = 2;
    -- All three statements are buffered, not executed yet
    ```
  </Step>
  
  <Step title="SELECT queries run immediately">
    `SELECT` queries are not buffered - they execute immediately even within a transaction:
    
    ```sql
    BEGIN;
    INSERT INTO users VALUES (1, 'Alice', 'alice@example.com');
    SELECT * FROM users WHERE id = 1;
    -- SELECT runs immediately; it won't see the buffered INSERT
    COMMIT;
    ```
    
    <Warning>
    `SELECT` queries inside a transaction do not see buffered changes until `COMMIT` is executed.
    </Warning>
  </Step>
  
  <Step title="COMMIT executes the buffer">
    When you execute `COMMIT`, all buffered statements are executed in order:
    
    ```sql
    BEGIN;
    INSERT INTO orders VALUES (1, 100, 1000.00);
    INSERT INTO order_items VALUES (1, 1, 5, 200.00);
    COMMIT;  -- Both inserts execute now
    ```
  </Step>
  
  <Step title="ROLLBACK discards the buffer">
    When you execute `ROLLBACK`, all buffered statements are discarded:
    
    ```sql
    BEGIN;
    DELETE FROM products WHERE price < 5.00;
    ROLLBACK;  -- DELETE is discarded, no rows deleted
    ```
  </Step>
</Steps>

## Current Limitations

<Warning>
Datacave's transaction implementation has important limitations:

- **No multi-statement atomicity**: If a buffered statement fails during COMMIT, previous statements may have already been applied. There is no all-or-nothing guarantee.
- **No isolation**: Concurrent transactions can see each other's uncommitted changes once they are written to storage.
- **No MVCC**: Datacave does not implement multi-version concurrency control.
- **SELECT doesn't see buffered changes**: Queries within a transaction do not reflect uncommitted inserts, updates, or deletes.
</Warning>

From the compatibility matrix:

> **Transactions**: `BEGIN`/`COMMIT`/`ROLLBACK` accepted; mutating statements buffered until COMMIT; no guaranteed multi-statement atomicity or isolation.

### What This Means

If you execute:

```sql
BEGIN;
INSERT INTO accounts VALUES (1, 'Alice', 1000.00);
INSERT INTO accounts VALUES (2, 'Bob', 500.00);
INSERT INTO accounts VALUES (1, 'Charlie', 750.00);  -- Duplicate primary key
COMMIT;
```

The first two inserts may succeed, and the third may fail. There is no rollback of the first two statements.

## Transaction Use Cases

### Grouping Related Inserts

Use transactions to group logically related inserts:

```sql
BEGIN;
INSERT INTO orders VALUES (101, 1, 1000.00, 1678901234);
INSERT INTO order_items VALUES (1, 101, 5, 200.00);
INSERT INTO order_items VALUES (2, 101, 3, 150.00);
COMMIT;
```

This groups the order and its items together, making the intent clear.

<Note>
While this groups the statements logically, remember that partial failures are possible. Implement error handling in your application.
</Note>

### Discarding Exploratory Changes

Use `ROLLBACK` to discard changes during exploration:

```sql
BEGIN;
DELETE FROM test_data WHERE category = 'obsolete';
SELECT COUNT(*) FROM test_data;  -- Check how many rows remain
ROLLBACK;  -- Discard the delete
```

### Batching Writes for Performance

Grouping multiple statements in a transaction can improve write throughput:

```sql
BEGIN;
INSERT INTO events VALUES (1, 100, 'login', 1678901234, 1.0);
INSERT INTO events VALUES (2, 100, 'click', 1678901235, 1.0);
INSERT INTO events VALUES (3, 100, 'logout', 1678901236, 1.0);
-- ... many more inserts ...
COMMIT;  -- All inserts executed together
```

Batching reduces per-statement overhead.

## Transaction Examples

### Using Transactions in psql

```sql
psql "host=127.0.0.1 port=5433 user=admin dbname=default"

default=> BEGIN;
BEGIN
default=> INSERT INTO users VALUES (1, 'Alice', 'alice@example.com');
INSERT 0 1
default=> INSERT INTO users VALUES (2, 'Bob', 'bob@example.com');
INSERT 0 1
default=> COMMIT;
COMMIT
```

### Using Transactions in Python (psycopg2)

```python
import psycopg2

conn = psycopg2.connect(
    host="127.0.0.1",
    port=5433,
    user="admin",
    password="change-me",
    dbname="default"
)

cur = conn.cursor()

try:
    # Begin transaction (implicit in psycopg2)
    cur.execute("BEGIN")
    
    cur.execute("INSERT INTO accounts VALUES (%s, %s, %s)",
                (1, "Alice", 1000.00))
    cur.execute("INSERT INTO accounts VALUES (%s, %s, %s)",
                (2, "Bob", 500.00))
    
    # Commit transaction
    cur.execute("COMMIT")
    print("Transaction committed")
except Exception as e:
    # Rollback on error
    cur.execute("ROLLBACK")
    print(f"Transaction rolled back: {e}")
finally:
    cur.close()
    conn.close()
```

<Note>
psycopg2 automatically starts a transaction for you. You can use `conn.commit()` and `conn.rollback()` instead of executing SQL commands.
</Note>

### Using Transactions in Node.js (pg)

```javascript
const { Client } = require('pg');

const client = new Client({
  host: '127.0.0.1',
  port: 5433,
  user: 'admin',
  password: 'change-me',
  database: 'default',
});

await client.connect();

try {
  await client.query('BEGIN');
  
  await client.query(
    'INSERT INTO accounts VALUES ($1, $2, $3)',
    [1, 'Alice', 1000.00]
  );
  await client.query(
    'INSERT INTO accounts VALUES ($1, $2, $3)',
    [2, 'Bob', 500.00]
  );
  
  await client.query('COMMIT');
  console.log('Transaction committed');
} catch (e) {
  await client.query('ROLLBACK');
  console.error('Transaction rolled back:', e);
} finally {
  await client.end();
}
```

### Using Transactions in Rust (tokio-postgres)

```rust
use tokio_postgres::{NoTls, Error};

#[tokio::main]
async fn main() -> Result<(), Error> {
    let (client, connection) = tokio_postgres::connect(
        "host=127.0.0.1 port=5433 user=admin password=change-me dbname=default",
        NoTls,
    ).await?;

    tokio::spawn(async move {
        if let Err(e) = connection.await {
            eprintln!("connection error: {}", e);
        }
    });

    // Begin transaction
    client.execute("BEGIN", &[]).await?;

    match client.execute(
        "INSERT INTO accounts VALUES ($1, $2, $3)",
        &[&1i64, &"Alice", &1000.0f64],
    ).await {
        Ok(_) => {
            client.execute(
                "INSERT INTO accounts VALUES ($1, $2, $3)",
                &[&2i64, &"Bob", &500.0f64],
            ).await?;
            
            // Commit transaction
            client.execute("COMMIT", &[]).await?;
            println!("Transaction committed");
        }
        Err(e) => {
            // Rollback transaction
            client.execute("ROLLBACK", &[]).await?;
            eprintln!("Transaction rolled back: {}", e);
        }
    }

    Ok(())
}
```

## Best Practices

<Steps>
  <Step title="Keep transactions short">
    Since there is no guaranteed isolation, keep transactions brief to minimize the window for conflicts:
    
    ```sql
    -- Good: short transaction
    BEGIN;
    INSERT INTO orders VALUES (101, 1, 1000.00, 1678901234);
    INSERT INTO order_items VALUES (1, 101, 5, 200.00);
    COMMIT;
    
    -- Avoid: long-running transaction
    BEGIN;
    -- many statements...
    -- complex queries...
    -- long delay...
    COMMIT;
    ```
  </Step>
  
  <Step title="Don't rely on atomicity">
    Since partial failures are possible, implement error handling and reconciliation logic in your application:
    
    ```python
    # Check for duplicate keys before inserting
    cur.execute("SELECT COUNT(*) FROM users WHERE id = %s", (user_id,))
    if cur.fetchone()[0] == 0:
        cur.execute("INSERT INTO users VALUES (%s, %s, %s)",
                    (user_id, username, email))
    ```
  </Step>
  
  <Step title="Batch writes when possible">
    Use transactions to batch multiple writes for better performance:
    
    ```javascript
    await client.query('BEGIN');
    for (const event of events) {
      await client.query(
        'INSERT INTO events VALUES ($1, $2, $3, $4, $5)',
        [event.id, event.user_id, event.type, event.timestamp, event.value]
      );
    }
    await client.query('COMMIT');
    ```
  </Step>
  
  <Step title="Use ROLLBACK for cleanup">
    Use `ROLLBACK` to discard test data or exploratory changes:
    
    ```sql
    BEGIN;
    INSERT INTO test_users VALUES (999, 'Test User', 'test@example.com');
    -- Run some tests...
    ROLLBACK;  -- Clean up test data
    ```
  </Step>
</Steps>

## Error Handling

Always handle transaction errors gracefully:

### Python Example

```python
try:
    cur.execute("BEGIN")
    
    # Your statements here
    cur.execute("INSERT INTO users VALUES (%s, %s, %s)",
                (1, "Alice", "alice@example.com"))
    cur.execute("INSERT INTO orders VALUES (%s, %s, %s)",
                (101, 1, 1000.00))
    
    cur.execute("COMMIT")
except psycopg2.Error as e:
    # Rollback on any error
    cur.execute("ROLLBACK")
    print(f"Error: {e}")
    # Implement retry logic or compensating actions
```

### Node.js Example

```javascript

}
```

## Connection State

Datacave tracks transaction state via the PostgreSQL `ReadyForQuery` message:

| State | Code | Description |
|-------|------|-------------|
| Idle | `I` | Not in a transaction |
| InTransaction | `T` | Active transaction (after BEGIN) |
| Error | `E` | Error state (requires ROLLBACK) |

Most client libraries handle this automatically, but it's useful to understand the state machine.

## Future Improvements

Datacave's transaction support will improve in future releases:

- Multi-statement atomicity (all-or-nothing commits)
- Isolation levels (READ COMMITTED, SERIALIZABLE)
- MVCC (multi-version concurrency control)
- Savepoints (partial rollback)
