## Overview

Datacave supports distributed deployments with data sharding and replication for horizontal scalability and high availability. The cluster architecture consists of:

- **Sharding**: Data partitioning across multiple shards using consistent hashing
- **Replication**: Each shard can have multiple replicas for fault tolerance
- **Failover**: Automatic promotion of replicas when primary shards fail

## Cluster Architecture

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Node 1    │     │   Node 2    │     │   Node 3    │
│             │     │             │     │             │
│  Shard 0    │────▶│  Shard 0'   │────▶│  Shard 0''  │
│  Shard 1    │     │  Shard 1'   │     │  Shard 1''  │
│             │     │             │     │             │
└─────────────┘     └─────────────┘     └─────────────┘
   Primary            Replica 1           Replica 2
```

With `shard_count = 2` and `replication_factor = 3`, each shard has 2 additional replicas distributed across the cluster.

## Sharding Configuration

### Shard Count

Configure the number of shards in your `config.toml`:

```toml
[sharding]
shard_count = 4
```

<ParamField path="shard_count" type="integer" default="4">
  Number of shards for data partitioning. Data is distributed using consistent hashing based on the primary key.
</ParamField>

### Choosing Shard Count

<AccordionGroup>
  <Accordion title="Small datasets (< 100 GB)">
    **Recommended**: 2-4 shards
    
    Provides parallelism without excessive overhead. Good for development and small production deployments.
  </Accordion>
  
  <Accordion title="Medium datasets (100 GB - 1 TB)">
    **Recommended**: 8-16 shards
    
    Balances parallelism and manageability. Each shard handles 62.5-125 GB with 16 shards.
  </Accordion>
  
  <Accordion title="Large datasets (> 1 TB)">
    **Recommended**: 32-64 shards
    
    High parallelism for large-scale deployments. Each shard handles 15.6-31.25 GB with 64 shards.
  </Accordion>
</AccordionGroup>

<Warning>
  **Shard count cannot be changed after initialization** without data migration. Plan capacity carefully before deployment.
</Warning>

### Shard Distribution

Datacave uses consistent hashing to distribute data across shards:

1. Primary key is hashed using a consistent hash function
2. Hash value determines the target shard
3. All rows with the same primary key reside on the same shard

This ensures:
- Even distribution across shards
- Consistent routing for reads and writes
- Minimal resharding when adding nodes

## Replication Configuration

### Replication Factor

Configure the number of replicas in `config.toml`:

```toml
[cluster]
replication_factor = 3
```

<ParamField path="replication_factor" type="integer" default="3">
  Total number of copies for each shard (including the primary). A factor of 3 means 1 primary + 2 replicas.
</ParamField>

### Replication Strategies

<Tabs>
  <Tab title="No Replication (RF=1)">
    ```toml
    [cluster]
    replication_factor = 1
    ```
    
    **Use case**: Development, testing, ephemeral data
    
    **Pros**:
    - Minimal storage overhead
    - Fastest writes
    - Simple deployment
    
    **Cons**:
    - No fault tolerance
    - Data loss if node fails
    - No high availability
  </Tab>
  
  <Tab title="Standard (RF=3)">
    ```toml
    [cluster]
    replication_factor = 3
    ```
    
    **Use case**: Production deployments, standard HA requirements
    
    **Pros**:
    - Tolerates 1 node failure
    - Good balance of durability and cost
    - Read load distribution across replicas
    
    **Cons**:
    - 3x storage cost
    - Moderate write overhead
  </Tab>
  
  <Tab title="High Availability (RF=5)">
    ```toml
    [cluster]
    replication_factor = 5
    ```
    
    **Use case**: Mission-critical data, multi-region deployments
    
    **Pros**:
    - Tolerates 2 node failures
    - High availability
    - Maximum durability
    
    **Cons**:
    - 5x storage cost
    - Higher write latency
    - More network bandwidth
  </Tab>
</Tabs>

## Cluster Setup

### Single-Node Cluster

```toml
# config.toml
[server]
listen_addr = "127.0.0.1:5433"

[sharding]
shard_count = 4

[cluster]
replication_factor = 1
```

Suitable for development and testing.

### Multi-Node Cluster

<Steps>
  <Step title="Configure Node 1 (Primary)">
    ```toml
    # node1-config.toml
    [server]
    listen_addr = "10.0.1.10:5433"
    
    [sharding]
    shard_count = 8
    
    [cluster]
    replication_factor = 3
    node_id = "node1"
    peers = ["10.0.1.11:5433", "10.0.1.12:5433"]
    ```
  </Step>
  
  <Step title="Configure Node 2 (Replica)">
    ```toml
    # node2-config.toml
    [server]
    listen_addr = "10.0.1.11:5433"
    
    [sharding]
    shard_count = 8
    
    [cluster]
    replication_factor = 3
    node_id = "node2"
    peers = ["10.0.1.10:5433", "10.0.1.12:5433"]
    ```
  </Step>
  
  <Step title="Configure Node 3 (Replica)">
    ```toml
    # node3-config.toml
    [server]
    listen_addr = "10.0.1.12:5433"
    
    [sharding]
    shard_count = 8
    
    [cluster]
    replication_factor = 3
    node_id = "node3"
    peers = ["10.0.1.10:5433", "10.0.1.11:5433"]
    ```
  </Step>
  
  <Step title="Start All Nodes">
    ```bash
    # On each node
    datacave-server serve --config /path/to/node-config.toml
    ```
  </Step>
</Steps>

<Note>
  The multi-node cluster configuration shown above uses example configuration that may not be fully implemented. Refer to the latest documentation for actual cluster setup procedures.
</Note>

## Failover and Recovery

### Automatic Failover

When a primary shard fails:

1. **Detection**: Cluster detects node failure via health checks
2. **Election**: Replicas elect a new primary using consensus
3. **Promotion**: Selected replica is promoted to primary
4. **Replication**: New primary begins serving reads and writes

Typical failover time: 5-30 seconds depending on cluster size and network latency.

### Manual Failover

Force failover for maintenance:

```bash
# Promote specific replica to primary
datacave-server cluster promote-replica --shard 0 --replica-id node2
```

### Recovery Procedures

<AccordionGroup>
  <Accordion title="Node Recovery After Crash">
    When a node recovers after a crash:
    
    1. Node rejoins the cluster
    2. Catches up with primary using WAL replay
    3. Once synchronized, becomes available replica
    
    ```bash
    # Check replication lag
    datacave-server cluster replication-status --shard 0
    ```
  </Accordion>
  
  <Accordion title="Add New Replica">
    Add a new replica to an existing shard:
    
    ```bash
    # Add replica
    datacave-server cluster add-replica --shard 0 --node-id node4
    
    # Monitor replication progress
    datacave-server cluster replication-status --shard 0
    ```
  </Accordion>
  
  <Accordion title="Remove Failed Replica">
    Remove a permanently failed replica:
    
    ```bash
    # Remove replica
    datacave-server cluster remove-replica --shard 0 --node-id node2
    ```
  </Accordion>
</AccordionGroup>

## Cluster Management

### Monitoring Cluster Health

```bash
# Overall cluster status
datacave-server cluster status

# Shard distribution
datacave-server cluster shard-distribution

# Replica health
datacave-server cluster replica-health
```

### Rebalancing Shards

When adding or removing nodes, rebalance data:

```bash
# Initiate rebalancing
datacave-server cluster rebalance

# Check rebalance progress
datacave-server cluster rebalance-status
```

<Warning>
  Rebalancing moves data between nodes and can impact performance. Schedule during low-traffic periods.
</Warning>

## Network Requirements

### Bandwidth

- **Replication traffic**: Proportional to write rate × (replication_factor - 1)
- **Recommendation**: 1 Gbps minimum, 10 Gbps for high-throughput deployments

### Latency

- **Intra-cluster**: < 5ms for optimal performance
- **Cross-region**: 50-100ms acceptable with tuning

### Ports

- **5433**: Database server (cluster communication)
- **9898**: Metrics and health checks

Ensure firewalls allow traffic between cluster nodes on these ports.

## Troubleshooting

<AccordionGroup>
  <Accordion title="Replica lag increasing">
    **Symptoms**: Replication lag continuously growing
    
    **Causes**:
    - High write rate exceeding replica throughput
    - Network congestion
    - Slow disk on replica
    
    **Solutions**:
    - Check network bandwidth: `iftop -i eth0`
    - Monitor disk I/O: `iostat -x 1`
    - Increase `memtable_max_bytes` to batch writes
    - Add more replicas to distribute load
  </Accordion>
  
  <Accordion title="Split-brain scenario">
    **Symptoms**: Multiple primaries for same shard
    
    **Causes**:
    - Network partition
    - Cluster membership inconsistency
    
    **Solutions**:
    - Check network connectivity between nodes
    - Verify cluster configuration is identical
    - Manually demote one primary: `datacave-server cluster demote --shard 0`
  </Accordion>
  
  <Accordion title="Node not joining cluster">
    **Symptoms**: New node fails to join cluster
    
    **Causes**:
    - Firewall blocking ports
    - Mismatched shard_count configuration
    - Clock skew between nodes
    
    **Solutions**:
    - Verify firewall rules: `nc -zv <peer-ip> 5433`
    - Ensure `shard_count` matches across all nodes
    - Synchronize clocks with NTP
  </Accordion>
</AccordionGroup>
